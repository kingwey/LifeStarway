# LifeStarway 项目深度分析报告

**分析日期**: 2026-07-15  
**分析范围**: 全量代码审查（后端 + 前端 + 基础设施）  

---

## 一、项目概况

**定位**: 数据沉淀迭代式 AI 全生命周期职业生涯规划系统  
**技术栈**: FastAPI + SQLAlchemy(async) + Vue3 + ECharts + Element Plus  
**核心模块**: 认证、人生档案、职业诊断、规划方案、人生星图、What-If 沙盒

---

## 二、架构现状评估

### ✅ 做得好的部分

| 维度 | 评价 |
|------|------|
| **模块分层** | API / Service / Model / Schema 四层清晰，职责分离 |
| **异步设计** | 全链路 async/await (FastAPI + SQLAlchemy async + httpx async) |
| **AI 集成** | 统一 `call_llm` 入口，支持多 Provider（dashscope/doubao/gemini）自动切换 |
| **降级处理** | Redis 不可用时自动降级；LLM 解析失败时保存原始文本 |
| **前端路由** | 路由守卫 + 懒加载，认证跳转逻辑完善 |
| **API 拦截器** | 自动携带 Token，401 自动跳登录 |

### 🔴 严重问题（P0 - 必须立即修复）

#### 1. 档案版本机制是假的

`backend/app/models/profile.py` 只有一条记录，更新时 `version += 1` 但旧版本数据直接覆盖，没有版本历史表。`get_profile_versions` 查出来始终只有一条。这导致版本对比功能完全无法使用。

**修复方案**: 需要新增 `ProfileVersion` 表，每次更新前将旧数据快照存入版本表。

#### 2. 文件上传依赖缺失

`backend/app/api/profile.py` PDF 解析依赖 `PyMuPDF(fitz)`、Word 解析依赖 `python-docx`，但这两个包不在 pyproject.toml 的依赖列表中，文件上传功能实际上不可用。

#### 3. 公开信息采集功能是半成品

`backend/app/services/scraper_service.py` 存在以下问题：
- `GITHUB_TOKEN` 在 config.py 中未定义，`hasattr(settings, 'GITHUB_TOKEN')` 始终为 False
- LinkedIn 采集只提取用户名，完全没有实际采集能力
- 博客解析用正则粗暴去 HTML 标签，会丢失所有结构化信息
- 没有 User-Agent，容易被反爬拦截
- Stars 字段错误地显示了 `public_gists`（公开Gist数）而非 stargazers_count
- 采集结果是拼接的原始文本，直接喂给 resume 解析 LLM，效果很差

#### 4. 职业诊断、规划生成没有异常处理

`backend/app/services/plan_service.py` 和 `backend/app/services/whatif_service.py` 中 `call_llm` 和 `parse_llm_*` 没有 try-catch，LLM 返回格式错误时直接 500。只有 `import_resume` 做了降级。

### 🟡 重要问题（P1 - 尽快修复）

#### 5. 数据模型设计缺陷

| 问题 | 位置 | 说明 |
|------|------|------|
| Profile 缺少唯一约束 | `backend/app/models/profile.py` | `user_id` 应该有 `unique=True`，否则可能产生重复档案 |
| JSON 字段无校验 | 多处 | `skills`, `dimensions`, `milestones` 等 JSON 字段没有结构约束，脏数据风险大 |
| Diagnosis 无外键约束行为 | `backend/app/models/diagnosis.py` | 删除 Profile 时关联的 Diagnosis 会报错（没有 ON DELETE CASCADE） |
| Plan 没有关联到用户当前档案版本 | `backend/app/models/plan.py` | 缺少 `profile_version` 字段，无法溯源 |

#### 6. JWT Token 过期时间硬编码不一致

`backend/app/utils/deps.py` 中默认过期时间写死为 `timedelta(minutes=15)`，但登录时用了 `settings.ACCESS_TOKEN_EXPIRE_MINUTES`(30分钟)，两套过期时间不一致。

#### 7. 前端大量空 catch

`frontend/src/views/Dashboard.vue` 和 `frontend/src/stores/user.js` 中大量使用 `catch {}` 空捕获，出错时用户无任何反馈，调试极其困难。

#### 8. 前端状态管理薄弱

- 诊断、规划、模拟数据没有在 Pinia store 中管理，每个页面单独请求，数据不共享
- 切换页面后重新加载，没有缓存
- Profile 更新后其他页面不知道，需要手动刷新

#### 9. LLM 调用层缺少限流和重试增强

`backend/app/ai/llm_client.py` 有重试但没有退避抖动（jitter），高并发时可能触发雪崩；没有请求限流；没有成本追踪。

#### 10. CORS 配置问题

`backend/.env` 中 `CORS_ORIGINS` 只有 5173 和 3000，但 Vite 可能分配其他端口（如之前出现过的 5175），应使用通配符或动态获取。

### 🟢 改进项（P2 - 体验优化）

| # | 问题 | 说明 |
|---|------|------|
| 11 | 全局日志中间件使用 BaseHTTPMiddleware | 存在已知性能问题，建议改用纯 ASGI 中间件 |
| 12 | Dockerfile 中 uv 使用方式不对 | 先 `pip install uv` 再 `uv pip install` 丢失了 uv 的速度优势，应直接用 `uv sync` |
| 13 | requirements.txt 和 pyproject.toml 不同步 | requirements.txt 缺少 redis, python-multipart, asyncpg, bcrypt, email-validator |
| 14 | alembic 迁移未实际使用 | 启动时用 `Base.metadata.create_all` 建表，alembic 配置形同虚设 |
| 15 | 前端没有 loading/error 状态组件 | 数据加载时无骨架屏，错误时无友好提示 |
| 16 | Dashboard 统计数据重复请求 | 进入页面同时请求 diagnosis 和 plans，starmap 又重新请求 plans，浪费 API |
| 17 | 密码强度无校验 | 注册时 `test123` 也能通过，没有密码复杂度要求 |
| 18 | 测试覆盖率低 | 现有测试只覆盖了基本 happy path，没有异常场景、边界条件、并发测试 |
| 19 | 没有分页 | `get_diagnoses`, `get_plans`, `get_simulations` 都是全量返回 |
| 20 | .env 中有真实 API Key 暴露在 git 历史中 | 建议立即轮换这些 Key |

## 四、分阶段优化推进方案

### 🔧 批次四：紧急修复（P0）

**目标**: 修复阻断性 Bug，确保核心流程稳定可用

1. **修复档案版本机制** — 新增 ProfileVersion 表，更新前自动快照
2. **补全依赖** — 将 PyMuPDF、python-docx 加入 pyproject.toml，确保文件上传可用
3. **修复 LLM 调用异常处理** — 所有 call_llm + parse 加 try-catch，返回友好错误
4. **修复 scraper_service 明显 bug** — Stars 统计错误、添加 User-Agent、补全 GITHUB_TOKEN config
5. **清理 requirements.txt** — 与 pyproject.toml 同步，或移除只保留 pyproject.toml
6. **修复 CORS 端口问题** — 开发环境允许 localhost 任意端口

### 🏗️ 批次五：架构加固（P1）

**目标**: 消除技术债务，提升可维护性

1. **数据库约束完善** — user_id 唯一约束、外键级联、添加必要索引
2. **统一 Pinia Store** — 新建 appStore 管理全局状态（profile/diagnosis/plans/simulations），页面间共享数据避免重复请求
3. **JWT 统一过期时间** — 移除 deps.py 中的硬编码 15 分钟，全部使用 settings
4. **前端错误处理** — 封装统一错误提示组件，替换空 catch；添加全局错误边界
5. **添加骨架屏/加载状态** — 列表和详情页加载时的视觉反馈
6. **完善 LLM 客户端** — 添加请求限流（令牌桶）、指数退避+抖动、成本统计日志
7. **修复日志中间件** — 替换 BaseHTTPMiddleware 为纯 ASGI 中间件
8. **添加请求参数校验** — 密码强度校验、邮箱格式二次校验

### 🚀 批次六：功能深化（P2）

**目标**: 完善产品核心体验

1. **版本对比功能真正可用** — 后端版本快照 + 前端 diff 可视化（类似 GitHub diff）
2. **规划详情与里程碑追踪** — 规划详情页、里程碑完成勾选、进度条
3. **诊断可视化增强** — 雷达图展示五维度、历史诊断趋势折线图
4. **公开信息采集增强** — 多源信息融合去重、Tavily 搜索补充、HTML 正文智能提取
5. **星图交互增强** — 拖拽位置持久化、星图导出PNG、多规划叠加
6. **分页支持** — diagnoses/plans/simulations 列表加分页
7. **用户设置页** — 修改昵称、修改密码、数据导出/删除

### 🌐 批次七：部署与上线准备

**目标**: 生产环境就绪

1. **Docker Compose 完善** — 修复 Dockerfile（uv sync）、添加健康检查重试、数据卷备份
2. **PostgreSQL 迁移** — 使用 Alembic 正式迁移，切换默认数据库为 PostgreSQL
3. **Redis 启动方案** — 提供 Windows 本地 Redis 安装说明或 docker-compose 一键启动脚本
4. **环境变量管理** — 区分 dev/staging/prod，真实 Key 从环境注入不提交
5. **Nginx 配置优化** — gzip 压缩、静态资源缓存、API 反向代理
6. **前端生产构建优化** — 代码分包、路由级别懒加载已做但检查 chunk 大小
7. **API 文档完善** — FastAPI 自带 Swagger 补充接口描述和示例

## 五、建议优先级排序

如果只能选最关键的先做，建议按以下顺序：

| 优先级 | 任务 | 原因 |
|--------|------|------|
| 1 | 修复 LLM 异常处理 | 当前诊断/规划/WhatIf 任何 LLM 格式错误都直接 500 |
| 2 | 补全文件上传依赖 | 文件上传是核心数据导入入口，当前不可用 |
| 3 | 修复版本机制 | 版本对比是产品卖点，当前是假的 |
| 4 | 统一状态管理 | 前端代码质量瓶颈，后续所有功能都依赖它 |
| 5 | 公开信息采集增强 | 这是用户刚要求的核心差异化功能，目前太简陋 |
